persona-feedback

合成ユーザーをサブエージェント化し、Webアプリに対するリアルなフィードバックを並列で集めるClaude Codeプラグイン。

Projectpersona-feedback
TypeClaude Code Plugin
DistributionGitHub Marketplace
LicenseMIT

Overview

本プラグインは「架空ユーザー（ペルソナ）をサブエージェントとして起動し、Playwright MCPを介して実際のWebアプリを操作させ、構造化されたフィードバックを得る」ためのClaude Codeプラグインである。開発者は実ユーザーテスト前に高速なフィードバックループを回せる。

Core Loop

  1. 定義：自然言語コンテキストからペルソナYAMLを生成（persona-builder）
  2. 実行：複数ペルソナをサブエージェントとして並列起動し、Playwright経由でアプリを操作（persona-tester）
  3. 集約：各ペルソナのフィードバックをマージし、不一致点をハイライトしたレポートを生成
Why this matters

LLMペルソナは「賢すぎ／協調的すぎ」という構造的問題を抱える。本プラグインは behavior_rules による意図的な能力制限と、ペルソナ間の不一致検出によってこの問題を緩和する。詳細は §16 を参照。

Goals & Non-Goals

Goals (v0.1)

  • Claude Codeのプラグインとしてgit経由でインストール可能
  • 2つのスキル（builder / tester）が独立して機能する
  • Playwright MCPを介したWebアプリ操作とスクリーンショット取得
  • 3体の組み込みペルソナと、自由なペルソナ生成
  • 並列実行（最低3ペルソナ同時）
  • Markdown / JSON両形式でのフィードバック出力
  • フィードバック観点（usability / bugs / accessibility 等）を実行時指定可能

Non-Goals (v0.1)

  • API / CLIアプリへの対応（v0.2以降）
  • ペルソナの永続メモリ（複数回テスト間の継続性。Agent SDK版で対応予定）
  • WebダッシュボードUI
  • 有償ホスティングサービス化
  • 実ユーザーテストの完全な代替（あくまで補助・前段）

Architecture

Key Properties

  • 分離されたコンテキスト：各サブエージェントは独自のシステムプロンプト・履歴を持ち、互いのフィードバックに汚染されない
  • 並列実行：メインエージェントはTask toolで複数サブエージェントを同時起動する
  • 共有MCP：Playwright MCPは全サブエージェントが利用できるが、ブラウザコンテキストはサブエージェント単位で分離
  • 構造化出力：各サブエージェントの最終出力はスキーマに従ったMarkdownまたはJSON

MCP Configuration

プラグインルートの .mcp.json にPlaywright MCPを定義する。プラグインインストール時にClaude Codeが自動でMCPサーバーを起動する。

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}

Playwright MCPは以下のツール群を提供する（実装AIは公式ドキュメントで最新のツール名を確認すること）：

  • browser_navigate — URLへ遷移
  • browser_snapshot — DOMのアクセシビリティスナップショット取得
  • browser_click / browser_type / browser_select — 操作
  • browser_take_screenshot — スクリーンショット保存
  • browser_wait_for — 待機
Decision required

Playwright MCPはNode.js環境を要する。READMEには「Node.js 18+ が必要」と明記すること。インストール手順に npx の初回実行コスト（数十秒）を注記する。

Skill: persona-builder

自然言語コンテキストを受け取り、構造化されたペルソナYAMLを生成するスキル。生成されたファイルはpersona-testerでそのまま消費可能。

SKILL.md（実装AI向けの仕様）

---
name: persona-builder
description: Use when the user wants to define, generate, or refine a synthetic
  user persona for testing. Triggers include "create a persona", "ペルソナ作って",
  "test user profile", or when the user describes a target user demographic.
  Outputs a YAML file conforming to the persona schema.
---

# persona-builder

ユーザーが提供するコンテキスト（業種・サービス・ターゲットユーザー像など）から、
テスト用ペルソナのYAMLファイルを生成する。

## 入力

ユーザーから以下のいずれか:
- 自由記述のコンテキスト（"ECサイト向けに3人のペルソナを"）
- 既存ペルソナYAMLへの修正指示
- ターゲットアプリのURL（任意 — Playwrightで実物を見て調整）

## 出力

`personas/<persona-id>.yaml` を作成する。スキーマは
`skills/persona-tester/schemas/persona.schema.json` に準拠すること。

## 生成手順

1. ユーザーのコンテキストを読み取り、必要なペルソナ数を確認
2. 各ペルソナについて以下を確定:
   - **demographics**: 年齢、職業、技術リテラシー
   - **context**: 利用デバイス、環境、目的
   - **personality**: 3〜5個の性格特性
   - **behavior_rules**: LLMの「賢すぎ問題」を抑制する明示ルール
   - **evaluation_focus**: そのペルソナが重視する評価軸
3. `templates/persona.template.yaml` を雛形として使用
4. 生成後、ユーザーに確認を求め、調整があれば再生成

## behavior_rules の設計指針 (CRITICAL)

LLMは本質的に「賢く・協調的に・推論で突破する」傾向がある。
リアルなユーザーをシミュレートするには、これを明示的に殺すルールが必要:

- 技術リテラシーが低いペルソナ: "専門用語が出たら推論せず『分からない』と判断"
- せっかちなペルソナ: "3秒以上の待機で離脱を検討"
- 不安症ペルソナ: "個人情報入力フォームで一度操作を停止し、安全性を確認"
- 慎重派ペルソナ: "エラーメッセージは読まず戻るボタンを優先"

これらは「能力の制限」であり、ペルソナの忠実度を担保する核心部分。

## 同梱ペルソナ

`personas/` 配下に最低3体のサンプルを置く:
- `tanaka-60s.yaml` — 技術リテラシー低、慎重派
- `gal-20s.yaml` — モバイル中心、せっかち、視覚重視
- `dev-engineer.yaml` — 高リテラシー、批判的、効率重視

Skill: persona-tester

指定されたペルソナ群を並列実行し、対象Webアプリを操作させてフィードバックを集約するスキル。

SKILL.md（実装AI向けの仕様）

---
name: persona-tester
description: Use when the user wants to test a web app with synthetic personas.
  Triggers include "test my app with personas", "ペルソナにテストさせて",
  "run persona feedback on <url>". Requires Playwright MCP. Spawns sub-agents
  per persona, executes tasks, returns aggregated feedback.
---

# persona-tester

複数の合成ペルソナをサブエージェントとして並列起動し、Webアプリを実際に操作させ、
構造化フィードバックを集約する。

## 入力

ユーザーから以下を取得（不足あれば質問）:
- **target**: テスト対象URL (必須)
- **personas**: ペルソナIDのリスト or YAMLファイルパス (必須)
- **task**: 実行させたいタスク (例: "新規登録してプロフィール画像をアップロード")
- **feedback_spec**:
    - focus: [usability, bugs, accessibility, conversion, copywriting]
    - severity_threshold: low | medium | high
    - output_format: markdown | json
- **parallel**: bool (default true)

## 実行フロー

1. **検証フェーズ**
   - 各ペルソナYAMLをスキーマ検証
   - targetがアクセス可能か Playwright で先に確認

2. **起動フェーズ**
   - ペルソナごとに Task tool でサブエージェントを起動
   - サブエージェントには以下を渡す:
     - `agents/persona-runner.md` のシステムプロンプト
     - 該当ペルソナのYAML（インラインで埋め込み）
     - target URL とタスク
     - feedback_spec
     - Playwright MCPツールへのアクセス

3. **実行フェーズ**（サブエージェント側）
   - browser_navigate で target にアクセス
   - ペルソナとして「自然に」タスクを試みる
   - 各ステップで:
     - 行動ログ（何を見て、何を考え、何をしたか）
     - スクリーンショット（重要なステップ）
     - 詰まり・違和感の記録
   - タスク完了 or 諦めポイントで終了
   - feedback.schema.json に準拠したJSONを返す

4. **集約フェーズ**
   - `scripts/aggregate.py` で全サブエージェント出力をマージ
   - **不一致点を抽出**:
     - 「全員が指摘した致命的問題」(severity = critical)
     - 「特定セグメントだけが詰まった箇所」(segment-specific)
     - 「ペルソナ間で評価が割れた要素」(controversial)
   - feedback_spec.output_format に従いMarkdown/JSON出力

## サブエージェントの数

- 既定で 3〜5 ペルソナまで並列
- 6体以上はシリアル実行も提案（コスト警告）

## エラーハンドリング

- 1ペルソナのセッション失敗で全体は止めない
- 失敗ペルソナの理由（タイムアウト/MCP接続失敗/etc）をレポートに含める

## 出力先

- Markdown: 標準出力 + `reports/<timestamp>-report.md`
- JSON: `reports/<timestamp>-report.json`
- スクリーンショット: `reports/<timestamp>/screenshots/`

Sub-agent Definition

agents/persona-runner.md はサブエージェントのシステムプロンプトを定義する。Claude CodeのTask toolはこのファイルを参照する。

---
name: persona-runner
description: A sub-agent that adopts a synthetic persona and performs UX testing
  on a target web app via Playwright MCP. Returns structured feedback.
tools:
  - mcp__playwright__browser_navigate
  - mcp__playwright__browser_snapshot
  - mcp__playwright__browser_click
  - mcp__playwright__browser_type
  - mcp__playwright__browser_take_screenshot
  - mcp__playwright__browser_wait_for
  - Write
---

あなたは合成ペルソナとしてWebアプリをテストするサブエージェントである。

# 守るべき原則

1. **あなたはペルソナそのものになりきる**。あなたの「Claude としての賢さ」は
   一切使わない。ペルソナの `behavior_rules` を文字通り守ること。

2. **推論で突破しない**。例えば技術リテラシー low のペルソナで「アップロード」
   という単語を見たら、知らないものは知らない。意味を推測しない。

3. **一人称で考える**。「私（このペルソナ）はこのボタンの意味が分からない。」
   と内的独白として記録する。

4. **諦めることを恐れない**。実ユーザーは詰まったら離脱する。ペルソナが
   詰まったら、無理に解決せず `outcome: abandoned` で終了してよい。

5. **証拠を残す**。重要な瞬間にはスクリーンショットを撮る。

# 実行手順

1. 受け取った persona YAML を熟読し、自分がそのペルソナだと内面化する
2. target URL に navigate
3. snapshot で画面構造を把握
4. ペルソナの第一印象を narrative に記録
5. task を実行する（ペルソナの能力範囲で）
6. 各ステップで findings を蓄積
7. タスク完了 / 諦め / エラーで終了
8. feedback.schema.json に準拠したJSONを最終出力

# 出力形式

最終メッセージで必ず以下を返す:

```json
{
  "persona_id": "...",
  "target": "...",
  "task": "...",
  "outcome": "completed | abandoned | blocked | error",
  "narrative": "私は...と感じた。...",
  "findings": [ ... ],
  "score": { "overall": 6.5, "would_recommend": false }
}
```

Critical design tension

LLMサブエージェントは本質的に「賢く・協調的・推論で突破する」傾向があり、初心者ユーザーの困惑を再現できない可能性がある。これは本プラグインの最大の構造的リスクである。対策として：

  • behavior_rules に明示的な能力制限を書く（§7参照）
  • サブエージェントシステムプロンプトで「Claudeとしての賢さを使うな」と強調（§11参照）
  • ペルソナ間の不一致を価値情報とする（全員「いい感じ」は信用しない）
  • v0.2でreal user testとの相関スタディを行う（ロードマップ）
